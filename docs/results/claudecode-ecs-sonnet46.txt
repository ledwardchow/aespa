[CRIT-05] Authorization Controlled by Client-Side Cookies
Location: All authenticated pages
Severity: Critical

After login, the server sets role-determining cookies in the HTTP response:

Set-Cookie: SysAdmin=true; path=/; HttpOnly
Set-Cookie: Manager=false; path=/; HttpOnly
Set-Cookie: login=true; path=/; HttpOnly
These cookies are marked HttpOnly but since the server trusts their values for access control, any user can escalate to SysAdmin by replaying a modified cookie. A regular user logging in can set SysAdmin=true in their cookie jar to access admin functions.

[HIGH-01] No HTTPS — Credentials Transmitted in Cleartext
Severity: High

The application runs exclusively over HTTP. All credentials, session tokens, and sensitive data are transmitted unencrypted. A network-adjacent attacker (or anyone who can see LAN traffic) can capture login credentials passively.

Additionally, session cookies lack the Secure flag, so they would be transmitted over HTTP even if HTTPS were added.

[HIGH-02] Reflected XSS in Support Page
Location: GET /ui/support.aspx?userid=
Severity: High

The userid parameter is reflected into a JavaScript variable using HTML encoding only. In a browser, HTML entities are decoded before JavaScript parsing, meaning &#39; becomes ' and breaks out of the string literal.

Payload: support.aspx?userid=test';alert(document.cookie);//

Server response (actual JS):

var selValue = 'test&#39;;alert(document.cookie);//';
A browser decodes &#39; → ' and executes alert(document.cookie).
